# Webnovel Learn

## Purpose

Use this Skill when the user gives feedback that should affect future planning, drafting, review, or querying. The goal is to preserve reusable writing experience in stable project files instead of relying on hidden chat memory.

## What To Record

Record only durable information:

- User preferences about pacing, point of view, tone, genre taste, chapter hooks, emotional intensity, and level of detail.
- Writing taboos such as unwanted tropes, forbidden phrasing, plot moves to avoid, or style patterns the user dislikes.
- Character setting corrections, including changed names, relationships, motivations, secrets, abilities, wounds, or status.
- Style lessons learned from revisions, reviews, or user comments.
- Long-term project memory such as unresolved major promises, recurring motifs, and slow-burn foreshadowing.

Do not record one-off task instructions unless the user says they should become a standing rule.

## Target Files

Write updates to the smallest relevant file:

- `设定集/写作偏好.md`: user preferences and reusable writing guidance.
- `设定集/风格禁区.md`: style taboos, banned moves, and phrasing to avoid.
- `人物状态/人物修正记录.md`: corrections to character canon or current state.
- `伏笔记录/长期伏笔.md`: long-running setup, delayed payoff, motifs, and open promises.

Create the file if it does not exist. Preserve existing notes and append dated entries unless the user explicitly asks for consolidation.

## Workflow

1. Identify whether the feedback is a preference, taboo, character correction, style lesson, or long-term memory.
2. Read only the target file needed for that category.
3. Append a concise entry with:
   - Date or relative session marker if known.
   - Source of the lesson, such as user feedback, review result, or chapter revision.
   - Concrete rule or memory to apply later.
   - Related chapter, character, or hook when relevant.
4. If feedback conflicts with existing records, point out the conflict and update only after the user confirms which version is canon.

## Output Style

After updating records, summarize:

- Which file was updated.
- What durable rule or memory was added.
- Any follow-up project files that may need synchronization.
